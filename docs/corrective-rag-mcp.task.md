## RAG Knowledge Base — MCP‑сервер с локальной LLM и LangGraph

### Описание

MCP‑сервер, который превращает локальную папку с документами в поисковую базу знаний. Разработчик подключает сервер к IDE, индексирует документы (проектная документация, внутренняя wiki, заметки) и задаёт вопросы — AI в IDE получает ответы, основанные на содержимом файлов. Внутри сервера — Corrective RAG‑пайплайн на LangGraph с локальной LLM через Ollama (без платных API).

### Результаты

- Git репозитарий с кодом и документацией
- MCP‑сервер с 4 инструментами (см. ниже)
- LangGraph‑граф с условными переходами и retry‑циклами (Corrective RAG)
- Docker Compose: сервер + Ollama + модель — запуск одной командой.
- Набор демо‑документов и тесты
- CI pipeline (lint + тесты; сборка Docker‑образа — опционально)

### Технологии

- Python, FastMCP
- LangGraph (оркестрация), LangChain (документ‑лоадеры, сплиттеры, `EnsembleRetriever` для гибридного поиска)
- Ollama + локальная маленькая LLM. (например, Phi‑3‑mini (3.8B) или Qwen2.5‑3B)
- ChromaDB — векторное хранилище (in‑process, без отдельного сервера)
- `rank_bm25` — sparse retrieval (BM25) для гибридного поиска
- Docker, Docker Compose

### Что нужно реализовать

#### MCP‑инструменты (4 инструмента)

| Инструмент | Вход | Что делает | Нужна LLM? |
|---|---|---|---|
| `index_folder` | путь к папке, glob‑паттерн | Сканирует файлы, разбивает на чанки, создаёт эмбеддинги, сохраняет в ChromaDB | Только эмбеддинги |
| `ask_question` | строка‑вопрос | Запускает полный RAG‑пайплайн (LangGraph) | Да — локальная LLM |
| `find_relevant_docs` | запрос, top_k | Возвращает ранжированные чанки без генерации ответа (гибридный поиск: BM25 + вектор → RRF) | BM25 + эмбеддинги |
| `index_status` | — | Статистика: кол‑во файлов, чанков, время последней индексации | Нет |

> **Требование к метаданным инструментов:** у каждого инструмента должно быть содержательное поле `description`, по которому хост‑агент (Copilot, Claude и др.) самостоятельно определяет, когда и зачем его вызывать — без подсказки пользователя. Это проверяется в шаге 5 процесса сдачи.

#### Поддерживаемые форматы документов

- `.md`, `.txt` — текстовые файлы
- `.py`, `.js`, `.ts` — код
- `.json`, `.yaml` — структурированные данные

#### LangGraph‑граф (Corrective RAG)

Поиск — гибридный: BM25 находит документы по точному совпадению ключевых слов, векторный поиск — по смыслу запроса. Результаты объединяются через Reciprocal Rank Fusion (RRF): итоговый ранг документа зависит от его позиций в обоих списках. Это позволяет одинаково хорошо отвечать на семантические вопросы («как работает авторизация») и точечные («где используется `TOKEN_EXPIRY_HOURS`»).

```
User Query → Rewrite Query → Retrieve (hybrid: BM25 + vector → RRF)
  → Grade Chunks (relevant? yes/no)
    → enough relevant → Generate Answer → Return answer + sources
    → too few relevant → Broaden Query → Retrieve (max 2 loops)
```

Grade Chunks — это «корректирующий» шаг, который дал название паттерну. LLM оценивает каждый retrieved чанк: релевантен ли он вопросу (yes/no). Если релевантных чанков недостаточно — граф расширяет запрос и повторяет поиск (до 2 раз). Это защищает от случаев, когда первый запрос сформулирован слишком узко или использует другую терминологию, чем в документах.

Подход к разбивке на чанки подбирайте исходя из природы данных (текст, код).

#### Дополнительные задания (необязательные)

- Поддержка внешней модели эмбеддингов через Ollama (например, `nomic-embed-text`) вместо дефолтной ChromaDB. Переключение через конфиг — без изменения остального кода.

## Критерии оценки

- **MCP‑сервер** — все 4 инструмента работают, корректные входы/выходы, обработка ошибок; описания (`description`) инструментов позволяют агенту выбрать нужный инструмент без подсказки пользователя
- **LangGraph‑граф** — Corrective RAG реализован: rewrite → retrieve (hybrid: BM25 + vector → RRF) → grade → generate. Условные переходы и retry‑циклы (расширение запроса, повторный поиск) работают
- **Индексация документов** — поддержка всех обязательных форматов, корректная разбивка на чанки, метаданные (источник, позиция) сохраняются
- **Тесты** — не менее 10 тестов: unit‑тесты графа с mock LLM, тесты индексера, e2e тест MCP‑инструментов
- **Инфраструктура** — Docker Compose запускается одной командой, CI pipeline (lint + тесты; сборка Docker‑образа — опционально)
- **Документы для индексации** - в комплекте. Объём - не менее 500кб.
- **Проверочные факты** предоставлены в README.md — это необычные, специфические факты (нестандартные числа, имена, даты и т.п.), намеренно добавленные в документы базы знаний и не встречающиеся в публичных источниках; они позволяют проверить, что RAG действительно читает именно вашу базу, а не отвечает из общих знаний модели
- **Качество кода** — читаемость, структура проекта, конфигурация вынесена, нет хардкода
- **AI‑процесс разработки** — REPORT.md содержит описание использованных инструментов и моделей, минимум один показательный пример промпта (удачный или неудачный) с кратким разбором
- **Документация** —
  - В корне проекта:
    - ARCHITECTURE.md: архитектура
    - README.md: инструкция по запуску, примеры использования
    - REPORT.md с вашим описанием истории создания проекта, с какими ключевыми проблемами вы столкнулись, как решали. Ваши удачные и неудачные идеи промтов. Вопросы. **Добавляйте туда данные по мере работы, не в самом конце**.
  - Есть конфиг-файл для подключения MCP сервера проекта в VSCode copilot

## Процесс сдачи

1. **Репозиторий** — студент предоставляет ссылку на GitHub‑репозиторий
2. **Запуск** — преподаватель выполняет:
   ```bash
   git clone <repo> && cd <repo>
   docker compose up
   ```
3. Преподаватель знакомится с README.md и следует дополнительным шагам настройки
4. Преподаватель подключает MCP сервер к своему агенту, используя предоставленный пример конфига MCP.
5. Преподаватель спрашивает у агента о фактах из базы студента
   - агент должен догадаться, что надо использовать подключенный MCP сервер (так что обратите внимание на его метаданные)
   - ответ должен содержать проверочные факты из базы знаний
6. **Проверка инструментов** — через MCP Inspector или подключение к IDE:
   - `index_status()` → пустой индекс
   - `index_folder("./sample_docs")` → индексация демо‑документов
   - `index_status()` → статистика (кол‑во файлов, чанков)
   - `ask_question("...")` → ответ с указанием источников
   - `find_relevant_docs("...")` → список релевантных чанков
7. **Код‑ревью** — просмотр структуры проекта, графа LangGraph, тестов
8. Ревью отчёта REPORT.md
9. **Демо (5 мин)** — студент показывает работу сервера в IDE, объясняет архитектуру графа (на спец занятии)
